# 07 — IBM Bob + GitHub Spec-Kit

## System of record

**IBM Bob** is the system of record and drives the GitHub **Spec-Kit** lifecycle. Claude (via
[../CLAUDE.md](../CLAUDE.md)) and Gemini (via [../GEMINI.md](../GEMINI.md)) are **parallel assistants**
that help draft and implement, but Bob owns the lifecycle and the `implement` step. See
[10-cross-ai-protocol](10-cross-ai-protocol.md) for how the three coordinate.

## The Spec-Kit lifecycle

```
/speckit.constitution  →  specify  →  clarify  →  plan  →  tasks  →  implement
```

1. **`/speckit.constitution`** — the versioned constitution: the non-negotiable rules and principles
   the project is built under. Amendments are versioned and recorded.
2. **`specify`** — write the feature specification (what and why, not how).
3. **`clarify`** — resolve ambiguities in the spec before any planning.
4. **`plan`** — turn the clarified spec into a technical plan (`specs/plan.md`).
5. **`tasks`** — decompose the plan into a checkbox task list (`specs/tasks.md`).
6. **`implement`** — **Bob owns this step.** Bob executes tasks against the plan; Claude/Gemini assist
   but Bob is authoritative for what lands.

## Generated vs. human-seeded artifacts

This distinction matters — do **not** hand-author generated files.

**Generated by running Bob / `specify init` (do NOT commit by hand):**

- `.bob/` — Bob's generated working directory and manifests.
- `.specify/scripts/` and Spec-Kit manifests — scaffolded by `specify init`.

These appear when you run the tooling. Editing them by hand causes drift and merge pain; re-run the
tooling instead.

**Human-seeded (authored and committed by the team):**

- The **constitution** content (via `/speckit.constitution`).
- `specs/` specifications, `specs/plan.md`, and `specs/tasks.md` (authored through the lifecycle).
- Shared-state files: `AGENTS.md` (rules), `STATUS.md` (live board).
- Assistant entry points: [../CLAUDE.md](../CLAUDE.md), [../GEMINI.md](../GEMINI.md).
- This `docs/` set.
